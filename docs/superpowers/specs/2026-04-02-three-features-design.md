# ATC Post-MVP: Three Transformative Features

**Date:** 2026-04-02
**Status:** Draft
**Scope:** Context Pane, Fleet Board, AI Sidekick

---

## Problem Statement

The ATC MVP successfully ingests agent messages, classifies them, and surfaces actionable items in an inbox. But the operator still opens Slack and Jira 5-10 times per day because:

1. **Insufficient context** — the inbox shows the last message, but the operator needs the conversation history and ticket context to act confidently
2. **No fleet overview** — there's no way to see all active work items at a glance, forcing the operator to ask their orchestrator agent "what do we have open?"
3. **No query interface** — finding a specific work item, filtering by agent, or asking "what happened overnight?" requires leaving the app

Three features address these gaps without turning ATC into a dashboard.

---

## Feature 1: Context Pane

### What it does

When the operator clicks/expands a work item card, a detail pane slides open showing everything needed to act — without leaving ATC.

### Sections (top to bottom)

1. **Work Item Header**
   - ID (linked to external tracker), title, current ATC status, external status from Jira
   - Assigned agent with avatar, last activity timestamp
   - Action buttons (same as card: Unblock, Done, Dismiss, Snooze)

2. **AI Summary** (the key differentiator)
   - 3-5 bullet points summarizing the conversation thread: what was attempted, what's the current state, what's blocking
   - Generated by a single LLM call using the thread's event history as context
   - Uses the same model provider as the classifier (operator's configured LLM)
   - Highlighted "relevant excerpt" — the specific part of the latest message that triggered the classification, extracted by the classifier's `reason` field

3. **Jira Context** (if a task adapter is connected and work item has a ticket ID)
   - Ticket description (first 500 chars)
   - Current Jira status, assignee, labels
   - Recent Jira comments (last 3-5)
   - Pulled from existing enrichment data + a fresh fetch on pane open

4. **Conversation Thread**
   - Scrollable list of messages from the thread, newest at bottom
   - Each message shows: author avatar + name, timestamp, message text
   - Messages that triggered non-noise classifications get a subtle left-border highlight
   - Agent messages vs operator messages visually distinct

5. **Suggested Replies**
   - 3-4 quick-reply chips based on the work item's status and common operator patterns
   - Examples: "Approved, proceed", "Hold — wait for my review", "Re-do with [constraint]", "Loop in @[agent]"
   - Configurable per-agent or per-status in `config/default.yaml` under a new `quickReplies` section
   - Clicking a chip populates the reply input; operator can edit before sending

### Architecture

**Backend changes:**
- New endpoint: `GET /api/work-item/:id/context` — returns summary, thread messages, enrichment, suggested replies in one payload
- New `Summarizer` module in `core/summarizer/` — takes thread events, calls the LLM with a summarization prompt, returns structured bullets
- Summarizer reuses the existing `ModelProvider` interface — no new LLM integration needed
- Summary is cached in a new `summaries` table (work_item_id, summary_text, generated_at) — regenerated if latest event is newer than cached summary
- Thread messages fetched via `PlatformAdapter.getThreadMessages(threadId)` — new method on the adapter interface
- Jira comments fetched via `TaskAdapter.getComments(workItemId)` — new method on the task adapter interface

**Frontend changes:**
- New `ContextPane` component — slides in from the right when a work item card is clicked
- WorkItemCard gets an `onClick` handler that opens the pane
- Pane width: ~50% of window, overlays the inbox list (does not push it)
- Close via X button, Escape key, or clicking outside
- Loading skeleton while the context endpoint returns

**New config:**
```yaml
quickReplies:
  # Status-based defaults
  blocked_on_human:
    - "Approved, proceed"
    - "Hold — waiting for my review"
    - "Re-do with the following constraint:"
  needs_decision:
    - "Go with option A"
    - "Go with option B"
    - "Need more info before deciding"
```

### What it does NOT do
- Does not show the full Slack UI or embed Slack
- Does not auto-send replies (operator always confirms)
- Does not cache summaries indefinitely — they're regenerated when new events arrive

## Feature 2: Fleet Board

### What it does

A second tab/view showing all active work items in a compact table, with anomaly detection badges.

### Layout

**Tab bar** at the top of the main content area, below the title bar:
- **Inbox** (current view — items needing attention)
- **Fleet** (new — all active work items)

The tab bar is minimal — two text labels, underline on active. No icons, no badges on the tabs themselves.

### Fleet Board columns

| Column | Content |
|--------|---------|
| Work Item | ID (linked) + title (truncated) |
| Agent | Avatar + name of last active agent |
| Status | ATC status badge (same component as inbox) |
| External | Jira/external status if available |
| Last Activity | Relative timestamp ("2h ago") |
| Anomalies | Badge icons (see below) |

### Anomaly detection

The fleet board surfaces exceptions via small badge icons on each row:

| Anomaly | Icon/Badge | Trigger |
|---------|-----------|---------|
| Stale | Clock icon, amber | No new events for >4 hours on an in_progress item |
| Silent agent | Warning icon, red | Agent's last_seen is >2 hours old while they have active work items |
| Status regression | Arrow-down icon, red | Status went from in_progress/completed back to blocked_on_human |
| Duplicate work | Link icon, purple | Two work items with the same title or ticket ID linked to different threads |

Anomaly thresholds are configurable in `config/default.yaml`:
```yaml
anomalies:
  staleThresholdHours: 4
  silentAgentThresholdHours: 2
```

### Search & Filter

A single search/filter bar above the table:
- **Text search**: filters by work item ID, title, or agent name (client-side filtering is fine for <500 items)
- **Status filter**: dropdown multi-select for ATC status categories
- **Agent filter**: dropdown multi-select populated from known agents
- **Anomaly filter**: toggle to show only items with anomalies

Filters are composable (AND logic). Filter state persists in URL query params so the view survives refresh.

### Architecture

**Backend changes:**
- New endpoint: `GET /api/fleet` — returns all non-completed work items with their latest event, agent, thread, and anomaly flags
- Anomaly detection runs server-side as a pure function over the work item + event data — no separate process or scheduled job
- New `detectAnomalies(workItem, events, agents, config)` function in `core/graph/anomalies.ts`

**Frontend changes:**
- New `FleetBoard` component with a table layout
- Tab bar added to `App.tsx` (new `View` variant: `"fleet"`)
- Clicking a row in the fleet board opens the Context Pane (shared with inbox)
- Responsive: on narrow windows, columns collapse to just Work Item + Status + Anomalies

**No new database tables** — anomaly detection is computed from existing data at query time.

## Feature 3: AI Sidekick

### What it does

A conversational query interface over the context graph. The operator types natural language questions and gets answers grounded in ATC's data.

### Interaction model

- **Trigger**: keyboard shortcut (Cmd+K) or a persistent input bar at the bottom of the screen
- **Panel**: slides up from the bottom, taking ~40% of screen height. Overlays the current view.
- **Conversation**: multi-turn within a session, but no persistence across app restarts (it's a query tool, not a chat history)
- **Close**: Escape, clicking outside, or Cmd+K again

### What it can answer

The sidekick has read-only access to the context graph. It can answer:

| Query type | Example | Data source |
|------------|---------|-------------|
| Agent lookup | "What is Byte working on?" | work_items + agents |
| Work item status | "What's the status of AI-382?" | work_items + events + enrichments |
| Time-based queries | "What happened overnight?" / "What completed today?" | events filtered by timestamp |
| Fleet summary | "How many items are blocked right now?" | work_items aggregate |
| Thread history | "Summarize the conversation about AI-500" | events for work item, passed to LLM |
| Anomaly queries | "Which agents have been quiet?" | agents + last_seen timestamps |

### Architecture

**How it works:**
1. Operator types a question
2. The question is sent to the LLM with a system prompt that describes the available "tools" (SQL queries over the context graph)
3. The LLM generates one or more tool calls (structured queries)
4. The backend executes them against SQLite and returns results
5. The LLM synthesizes a natural language answer from the results
6. The answer is displayed in the sidekick panel

This is a **tool-use pattern**, not RAG. The LLM doesn't search embeddings — it generates precise queries against known schema. This is more reliable for structured data and avoids the need for a vector store.

**Backend changes:**
- New endpoint: `POST /api/sidekick` — accepts `{ question: string, history: Message[] }`, returns `{ answer: string, sources: Source[] }`
- New `Sidekick` module in `core/sidekick/` with:
  - `system-prompt.md` — describes the context graph schema and available query tools
  - `tools.ts` — defines tool schemas: `query_work_items`, `query_events`, `query_agents`, `get_thread_summary`
  - `executor.ts` — executes tool calls against the ContextGraph (read-only, no raw SQL — uses existing ContextGraph methods + a few new query methods)
  - `index.ts` — orchestrates the question → tool calls → answer flow
- New ContextGraph query methods needed:
  - `searchWorkItems(query: string)` — text search over work_items.id, title
  - `getWorkItemsByAgent(agentId: string)` — work items where the latest event's agent matches
  - `getEventsSince(since: Date)` — all events after a timestamp
  - `getAgentByName(name: string)` — case-insensitive LIKE match on agent name
  - `getFleetStats()` — aggregate counts by status

**Frontend changes:**
- New `Sidekick` component — input bar + scrollable message list
- Messages alternate between operator questions and sidekick answers
- Answers can include inline work item IDs that link to the context pane
- Loading state: typing indicator while LLM processes
- Keyboard shortcut registration in `App.tsx`

**LLM usage:**
- Uses the same model provider as the classifier
- Each question = 1-3 LLM calls (initial reasoning + potential follow-up after tool results)
- Rate limited using the existing LLM rate limiter
- System prompt is ~1000 tokens; typical question + tool results + answer is ~2000-3000 tokens total

### What it does NOT do
- Does not take actions (no posting to Slack, no updating work items)
- Does not remember conversations across sessions
- Does not have access to Slack or Jira directly — only the context graph
- Does not stream responses in v1 (full response returned at once, streaming is a fast follow)

### Configuration
```yaml
sidekick:
  enabled: true
  # Max tool calls per question (prevents runaway loops)
  maxToolCalls: 5
  # Max conversation history to include (token budget)
  maxHistoryTurns: 10
```

## Shared Design Decisions

### UI Layout Changes

The current app has a single view with header + main content. The new layout:

```
+------------------------------------------+
| ATC                    [gear] [status]   |  <- existing header
+------------------------------------------+
| [Inbox]  [Fleet]                         |  <- new tab bar
+------------------------------------------+
|                          |               |
|   Work Item List         | Context Pane  |  <- pane slides from right
|   (inbox or fleet)       | (on click)    |
|                          |               |
+------------------------------------------+
| [Cmd+K] Ask ATC...                       |  <- sidekick trigger
+------------------------------------------+
```

The Context Pane is shared between Inbox and Fleet Board — clicking any work item opens it regardless of which tab you're on.

### LLM Budget

Both Context Pane (summarizer) and AI Sidekick use the operator's configured LLM. To keep costs manageable:
- Summaries are cached and only regenerated on new events
- Sidekick uses tool-use (small precise queries) rather than stuffing the full DB into context
- The existing rate limiter applies to all LLM calls (classifier + summarizer + sidekick share the budget)
- Consider bumping the default rate limit from 4/min to 8/min in config, or making it configurable per-feature

### No New External Dependencies

All three features build on existing infrastructure:
- Same LLM provider (model-agnostic, operator's choice)
- Same SQLite database (new tables only for summary cache)
- Same Hono server (new endpoints)
- No vector store, no new databases, no new services

### Mobile / Narrow Screens

ATC is a desktop app, but the web mode should degrade gracefully:
- Context Pane becomes full-width overlay on screens <1024px
- Fleet Board hides non-essential columns
- Sidekick panel becomes full-screen overlay

## Implementation Order

1. **Context Pane** — highest daily pain relief. Requires: new API endpoint, Summarizer module, PlatformAdapter.getThreadMessages, ContextPane component.
2. **Fleet Board** — replaces "what's open?" conversations. Requires: new API endpoint, anomaly detection, FleetBoard component, tab bar in App.tsx.
3. **AI Sidekick** — most ambitious, benefits from 1 & 2 being in place (sidekick can link to context pane, use anomaly data). Requires: Sidekick module with tool-use, new ContextGraph query methods, Sidekick component.

Each feature is independently shippable and useful on its own.

---

## Out of Scope

- Action-taking sidekick (posting to Slack, updating Jira) — v2
- Persistent sidekick conversation history — v2
- Custom anomaly rules — v2
- Streaming LLM responses — fast follow after sidekick ships
- Per-agent quick reply customization UI — config file is sufficient for now
